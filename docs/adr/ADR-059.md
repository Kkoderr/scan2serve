# ADR-059: Ngrok-Friendly Public Gateway URL (CORS + QR Base URL + CSRF)

**Date:** 2026-04-14
**Status:** Accepted

## Context
We want to connect a developer PC to **ngrok** so the product can be used on real devices (phones) and by collaborators.

Intent: **every browser/device request should hit the gateway via the ngrok public URL**, and the gateway should route `/` to web and `/api/*` to the API.

Today, local dev and demo workflows are awkward because:
- API CORS is locked to a single `CLIENT_URL` value.
- Table QR payload links are built using `CLIENT_URL`, which defaults to localhost and breaks on phones.
- CSRF relies on a cookie + header token; it must continue to work when accessed via ngrok (HTTPS) and via the gateway.

This makes it awkward to:
- Allow multiple/temporary public origins without code changes.
- Point QR links to a public site URL that may differ from the internal app/gateway origin.

## Decision
Introduce explicit environment configuration for:
- **CORS allowed origins** (including support for ngrok public URLs).
- **Public base URL used in QR payload links** for table QR codes.
- **CSRF safety under multiple origins** (avoid accidentally allowing unsafe origins to fetch CSRF + perform cookie-auth requests).

Proposed behavior:
- Add `CORS_ORIGINS` as a **comma-separated list** of allowed origins. When set, it controls CORS; otherwise fall back to `CLIENT_URL` to preserve existing behavior.
- Add `QR_BASE_URL` as a **single origin** (no path) used for QR payloads. When set, QR payloads use this origin; otherwise fall back to `CLIENT_URL`.
- Trim trailing slashes in the QR base URL to avoid double slashes in payload links.
- Ensure CSRF remains effective:
  - CSRF token cookie is still issued by `GET /api/auth/csrf`.
  - Mutating cookie-auth requests must still include `x-csrf-token` matching the `csrf_token` cookie.
  - Do not allow wildcard CORS origins when `credentials: true` is enabled.
  - No additional Origin enforcement is required beyond existing CSRF behavior + configured CORS origins (per answers).

## Consequences
- Developers can switch QR link bases and CORS origins without code edits.
- Production can keep strict CORS with a single configured origin.
- We must document the new env variables in `.env.example` and ensure defaults preserve current behavior.
- If we allow multiple public origins, we must be careful to not weaken CSRF by allowing untrusted origins to read CSRF responses.

## Questions & Answers

### Questions for User
- Q1: Confirm: `CORS_ORIGINS` should be a **comma-separated list** of exact origins (ex: `http://localhost:3000,https://xxxx.ngrok-free.app`)?
- Q2: Confirm: `QR_BASE_URL` should be **origin only** (no path), e.g. `https://xxxx.ngrok-free.app`?
- Q3: Confirm: `QR_BASE_URL` should only apply when explicitly set (otherwise fallback to `CLIENT_URL`)?
- Q4: For ngrok, do you want to manually paste the new ngrok URL into `.env` each session, or do you want to support an allow-pattern like `https://*.ngrok-free.app`?
  - Note: pattern support requires a custom CORS origin matcher (exact list-only is simpler and safer).
- Q5: CSRF hardening: do you want the API to **also** enforce an `Origin` allow-list for mutating cookie-auth requests (in addition to the CSRF cookie+header match)?
  - This is mostly defense-in-depth, but it prevents mistakes like accidentally allowing attacker origins via CORS.
- Q6: Should `CLIENT_URL` remain `http://localhost:3000` in compose (gateway origin), with ngrok handled via `QR_BASE_URL` + `CORS_ORIGINS`; or do you want `CLIENT_URL` itself to be the ngrok URL when running tunnels?
