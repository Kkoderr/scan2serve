# ADR-058: Configurable CORS and QR Base URL via Environment

**Date:** 2026-04-14
**Status:** Proposed

## Context
Local development and demo workflows often require exposing the app through a public tunnel (e.g., ngrok). Today:
- CORS is locked to `CLIENT_URL` in the API.
- QR payload links for table QR codes use `CLIENT_URL` as the base URL.

This makes it awkward to:
- Allow multiple/temporary public origins without code changes.
- Point QR links to a public site URL that may differ from the internal app/gateway origin.

## Decision
Introduce explicit environment configuration for:
- **CORS allowed origins** (including support for ngrok public URLs).
- **Public base URL used in QR payload links** for table QR codes.

Proposed behavior:
- Add an env var (name TBD) for CORS origins. When set, it controls CORS; otherwise fall back to `CLIENT_URL` to preserve existing behavior.
- Add an env var (name TBD) for QR base URL. When set, QR payloads use this origin; otherwise fall back to `CLIENT_URL`.
- Trim trailing slashes in the QR base URL to avoid double slashes in payload links.

## Consequences
- Developers can switch QR link bases and CORS origins without code edits.
- Production can keep strict CORS with a single configured origin.
- We must document the new env variables in `.env.example` and ensure defaults preserve current behavior.

## Questions & Answers

### Questions for User
- Q1: Should CORS be configured as a **single** origin (e.g., `CORS_ORIGIN`) or a **comma-separated list** (e.g., `CORS_ORIGINS`)?
- Q2: For QR links, should we use a **public site** origin only (e.g., `https://scan2serve.com`) or support a **full base URL with path** (e.g., `https://scan2serve.com/public`)?
- Q3: Do you want the QR base URL env var to **override** `CLIENT_URL` entirely, or should it only be used if explicitly set (fallback remains `CLIENT_URL`)?

### Answers (to be filled by user)
- A1: comma separated
- A2: origin only
- A3: it should only be used if explicitly set (fallback remains `CLIENT_URL`)?
